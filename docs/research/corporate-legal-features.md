# 기업법무 관점 법순이 기능 확장 연구

> **작성일**: 2025-12-27
> **상태**: 연구 완료, 이슈 등록 완료
> **관련 Epic**: #45

---

## 1. 연구 배경

기업법무(인하우스) 변호사 관점에서 **서버 인프라 없이** 구현 가능한 기능 영역을 탐색하고, 현재 법순이가 지원하는 기능과의 교집합을 분석하여 추가 기능을 도출함.

### 연구 방법
- 기업법무 변호사 업무 영역 분석
- 현재 법순이 기능 Gap 분석
- 서버 없이 활용 가능한 공개 API/데이터 조사
- 기능 아이디어 브레인스토밍 및 비판적 평가
- 리걸테크 시장 동향 조사

## 2. 기업법무 핵심 Pain Points

### 2.1 웹 조사 결과

| Pain Point | 설명 | 법순이 해결 가능성 |
|------------|------|------------------|
| **반복 질의 대응** | 여러 부서에서 같은 법률 질문 반복 | ⭐ 높음 |
| **광범위한 업무** | 계약, 송무, 컴플라이언스, 주총 등 전부 커버 | △ 부분 |
| **규제 변화 추적** | 중대재해법, 공정위 등 신규 리스크 증가 | ⭐ 높음 |
| **실무 교육 부족** | 법령 찾고 분석하는 실질적 가이드 부재 | ⭐ 높음 |
| **조직 내 위상** | 비변호사 상사에게 법적 리스크 설명 어려움 | △ 간접 지원 |

### 2.2 주요 업무 영역

1. 계약서 검토/작성
2. 법률 자문 회신 (사업부서 질의 대응)
3. 규제 컴플라이언스 모니터링
4. 소송/분쟁 관리
5. 이사회/주총 문서 준비
6. 정기 규제 보고
7. 내부 정책/가이드라인 업데이트
8. M&A Due Diligence
9. 해외 진출 검토
10. 신규 사업 법률 검토

## 3. 현재 법순이 기능 vs Gap

### 3.1 현재 지원 기능

**체크리스트 7종:**
- `startup` - 스타트업 설립
- `privacy_compliance` - 개인정보처리자
- `fair_trade` - 공정거래
- `contract_review` - 계약서 검토
- `labor_hr` - 노동/인사
- `serious_accident` - 중대재해
- `investment_due_diligence` - 투자 DD

**핵심 기능:**
- 법령 검색/다운로드 (법률, 시행령, 시행규칙, 행정규칙)
- 판례 검색
- 국회 의안 추적
- 법령 개정 모니터링
- 정책 동향 (부처 RSS, 법령해석례)
- 계약서 검토 보조 (`clause_references.yaml`)
- 법률용어 사전 (`legal_terms.yaml`)
- 해외 진출 가이드

### 3.2 Gap 분석

| 기업법무 영역 | 현재 지원 | Gap |
|-------------|---------|-----|
| 계약서 검토 | △ 부분 | 조항별 리스크 스코어링 없음 |
| 법률 자문 | ○ 법령 검색 | 자문 템플릿/포맷 없음 |
| 규제 모니터링 | ○ 개정/RSS | **업종별 규제 매핑 없음** |
| 이사회 문서 | ✗ | **상법 관련 조문 연결 없음** |
| 정기 보고 | ✗ | **법정 의무 캘린더 없음** |
| 내부 정책 | ✗ | **정책 템플릿 없음** |
| 신규 사업 검토 | △ startup | **업종별 규제 매핑 부족** |

## 6. 우선순위별 기능 상세

### Tier 1: 즉시 구현 가치 (YAML 기반)

#### 6.1 법정 의무 캘린더 `compliance_calendar.yaml`

**목적**: 기업이 반드시 준수해야 하는 법정 의무 일정 관리

```yaml
# 예시 구조
quarterly:
  - name: "공시대상기업집단 현황 공시"
    law: "독점규제법 시행령 제4조의2"
    authority: "공정거래위원회"
    deadline: "분기말 후 30일"
    penalty: "과태료 1억원 이하"
    applies_to: ["대기업집단"]

annual:
  - name: "정기 주주총회"
    law: "상법 제365조"
    deadline: "결산기 후 3개월 내"
    applies_to: ["주식회사"]
```

**가치**: 놓치면 과태료/제재, 기업법무 필수 기능

#### 6.2 업종별 규제 맵 `regulatory_map.yaml`

**목적**: "이 사업 하려면 어떤 법령 확인해야 해요?" 즉시 답변

```yaml
fintech:
  display_name: "핀테크/전자금융"
  primary_laws:
    - name: "전자금융거래법"
      key_articles: ["제28조(등록)", "제39조(과징금)"]
    - name: "신용정보법"
  licenses:
    - type: "전자금융업 등록/허가"
      authority: "금융위원회"
  regulatory_bodies: ["금융위원회", "금융감독원"]
  common_issues:
    - "개인(신용)정보 제3자 제공 동의"
```

**가치**: 사업부 질문 1분 내 초벌 답변

#### 6.3 허가/신고 매핑 `permits.yaml`

**목적**: 사업 행위별 필요한 인허가 확인

```yaml
online_business:
  통신판매업:
    type: "신고"
    authority: "시군구청"
    requirements: ["사업자등록증"]
    processing_time: "즉시~3일"
```

### Tier 2: 중기 구현

- **컴플라이언스 정책 템플릿** - 직장내괴롭힘, 개인정보, 공정거래
- **위험 신호 탐지기** - 계약서 위험 조항 경고 (`clause_references.yaml` 확장)
- **상법 연동 템플릿** - 이사회/주총 관련 상법 조문 연결

### Tier 3: 장기/실험

- DART 공시 연동
- 공정위 심결례 연동

## 7. 데이터 유지보수 전략 ⭐ CRITICAL

### 7.1 문제 인식

YAML 기반 정적 데이터의 한계:
- 법령 개정 시 데이터 outdated
- 수동 업데이트 시 누락 위험
- 잘못된 정보로 인한 법적 리스크

### 7.2 자동화/반자동화 전략

#### Option A: 완전 자동화
```yaml
# GitHub Actions - 월간 법령 변경 감지
on:
  schedule:
    - cron: '0 9 1 * *'  # 매월 1일 오전 9시

jobs:
  check-law-updates:
    steps:
      - name: Fetch recent law changes
        run: python scripts/fetch_law.py recent --days 30
      - name: Compare with YAML data
        run: python scripts/validate_yaml_data.py
      - name: Create issue if outdated
        if: steps.compare.outputs.outdated == 'true'
```

#### Option B: 반자동화 (권장)
```
[월간/분기별 점검 프로세스]
1. GitHub Actions가 법령 변경 감지
2. 자동으로 Issue 생성 (라벨: data-update)
3. 담당자가 수동 확인 후 YAML 업데이트
4. PR 생성 → CI 검증 → 머지
```

#### Option C: 커뮤니티 기여
- 업종별 전문가가 자신의 영역 업데이트
- CONTRIBUTING.md에 데이터 업데이트 가이드
- Good first issue로 표시

### 7.3 데이터 품질 보장

```yaml
# 각 데이터 항목에 메타데이터 포함
- name: "전자금융거래법"
  last_verified: "2025-12-27"
  verified_by: "sjlee"
  law_version: "법률 제19973호, 2024.1.16."
  confidence: "high"  # high/medium/low
```

### 7.4 면책조항 강화

```markdown
> ⚠️ **주의**: 이 데이터는 참고용이며, 법령 개정으로 인해
> 최신 정보와 다를 수 있습니다. 중요한 결정 전 반드시
> 법령정보센터(law.go.kr)에서 현행 법령을 확인하세요.
> 마지막 검증일: {last_verified}
```
